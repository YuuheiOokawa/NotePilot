## 1. プロダクト概要

### 1.1 プロダクト名

**Note Auto Creator** — note記事作成・販売支援アプリ

### 1.2 目的

noteで公開する無料記事・有料記事を、AIがほぼ自動で作成し、**ユーザーが最終確認・承認したうえで**投稿できる状態にするアプリ。記事のアイデア出しから執筆・有料設計・販売文作成までを支援し、ユーザーの作業を「確認・修正・承認」に集中させる。

### 1.3 解決する課題

| 課題 | 本アプリの解決策 |
|---|---|
| 記事テーマが思いつかない | AIによる売れ筋・得意分野ベースのテーマ提案 |
| 執筆に時間がかかる | タイトル〜本文〜まとめまでの自動生成 |
| 有料記事の設計が難しい | 無料/有料の切り分け・価格・構成の提案 |
| 記事の管理が煩雑 | ステータス管理による一元的なワークフロー |
| 売上の振り返りができない | 記事ごとの売上メモ・記録 |

### 1.4 ターゲットユーザー

- noteで無料・有料記事を発信したい個人（副業層・情報発信初心者〜中級者）
- 特にIT・転職・資格・副業・学習系の発信者
- MVPでは**シングルユーザー（本人利用）**を想定。将来的にマルチユーザー化可能な設計とする。

## 2. 基本方針・制約（最重要ルール）

これらは全機能に優先する制約であり、実装・レビュー時に必ず遵守を確認する。

| # | ルール | 実装上の意味 |
|---|---|---|
| R1 | AIが勝手に投稿・販売開始しない | 投稿・公開に相当する処理は存在しない。「posted」ステータスはユーザーが手動で記録するもの |
| R2 | 投稿・有料設定・価格設定は必ずユーザー承認制 | `review → approved` の遷移はユーザー操作のみ。AI・バッチ処理からは遷移不可 |
| R3 | noteの利用規約に反する自動操作をしない | スクレイピング・自動ログイン・自動投稿は実装しない |
| R4 | MVPはnote API連携なし | 記事本文を**コピーできる形（コピー用プレビュー）**で提供 |
| R5 | 将来の投稿連携を見据えた設計 | 記事データ構造・ステータス設計を連携可能な形にしておく（外部連携用フィールドの予約等） |

## 3. 機能要件

機能IDは `F-カテゴリ番号-連番`。優先度: **Must**（MVP必須）/ **Should**（MVPに入れたい）/ **Later**(将来拡張)。

### 3.1 記事テーマ提案（F-01）

| ID | 機能 | 内容 | 優先度 |
|---|---|---|---|
| F-01-1 | テーマ提案生成 | AIがテーマ候補を複数（5〜10件）提案する | Must |
| F-01-2 | 提案カテゴリ指定 | 「売れやすい」「無料向け」「有料向け」「経験を活かす」「IT・転職・資格・副業・学習」の観点を指定して提案させる | Must |
| F-01-3 | プロフィール反映 | 設定画面に登録したユーザーの経験・得意分野を提案に反映する | Should |
| F-01-4 | テーマ保存 | 気に入ったテーマを `article_ideas` として保存する（ステータス: idea） | Must |
| F-01-5 | テーマから記事作成 | 保存したテーマを起点に記事生成へ進める | Must |

### 3.2 記事自動生成（F-02）

| ID | 機能 | 内容 | 優先度 |
|---|---|---|---|
| F-02-1 | 記事一括生成 | テーマ・記事種別（無料/有料）・トーン等を入力し、記事全体を生成する | Must |
| F-02-2 | 生成要素 | タイトル / 導入文 / 目次 / 本文 / まとめ / CTA / ハッシュタグ / サムネイル文言 を生成する | Must |
| F-02-3 | 有料記事の切り分け | 有料記事の場合、無料公開部分と有料部分の境界を提案・生成する | Must |
| F-02-4 | セクション単位管理 | 記事は `article_sections` としてセクション単位で保持し、部分的な再生成・編集を可能にする | Must |
| F-02-5 | 手動編集 | 生成結果の全要素をユーザーが編集できる | Must |
| F-02-6 | セクション再生成 | 特定セクションのみAIに再生成させる | Should |
| F-02-7 | 生成ログ | AI生成の入出力を `ai_generation_logs` に記録する（コスト・品質改善用） | Should |

### 3.3 無料・有料記事管理（F-03）

| ID | 機能 | 内容 | 優先度 |
|---|---|---|---|
| F-03-1 | 記事一覧 | 記事種別（無料/有料）・ステータスで絞り込める一覧 | Must |
| F-03-2 | 記事詳細 | 記事の全要素・ステータス・履歴を表示 | Must |
| F-03-3 | ステータス管理 | 下記 3.5 のワークフローに沿った状態管理 | Must |
| F-03-4 | 下書き保存 | 生成途中・編集途中の記事を下書きとして保存 | Must |
| F-03-5 | 記事削除・アーカイブ | 不要な記事をアーカイブ（archived）にする | Must |

### 3.4 有料記事販売支援（F-04）

| ID | 機能 | 内容 | 優先度 |
|---|---|---|---|
| F-04-1 | 販売設計提案 | AIが価格・販売タイトル・無料公開範囲・有料部分の価値訴求・読者ターゲット・購入されやすい構成・宣伝文を提案する | Must |
| F-04-2 | 販売設定編集 | 提案内容をユーザーが編集し、記事に紐づけて保存する | Must |
| F-04-3 | 販売設定の承認必須 | 価格を含む販売設定は承認（approved）前に必ずユーザー確認を挟む（R2） | Must |

### 3.5 承認ワークフロー（F-05）

記事ステータスは以下の7状態。**太字の遷移はユーザー操作でのみ実行可能。**

```
idea → draft → review → **approved** → copied → **posted** → archived
                  ↓ 差し戻し
               draft（改善中）
```

| ステータス | 意味 | 遷移トリガー |
|---|---|---|
| idea | テーマ保存済み・未執筆 | テーマ保存時 |
| draft | 執筆中・改善中 | 記事生成時 / reviewからの差し戻し |
| review | 確認待ち（承認リクエスト中） | ユーザーが「確認に出す」操作 |
| approved | 承認済み・投稿可能 | **ユーザーの承認操作のみ** |
| copied | コピー用プレビューからコピー済み | コピー操作時に自動記録 |
| posted | noteに投稿済み（ユーザーの自己申告） | **ユーザーの手動記録のみ** |
| archived | アーカイブ | ユーザー操作 |

| ID | 機能 | 内容 | 優先度 |
|---|---|---|---|
| F-05-1 | 承認リクエスト | draft → review の申請を `approval_requests` に記録 | Must |
| F-05-2 | 承認待ち一覧 | review 状態の記事を一覧表示 | Must |
| F-05-3 | 承認・差し戻し | ユーザーが承認（approved）または差し戻し（draft）を行う。差し戻し時はコメントを残せる | Must |
| F-05-4 | 遷移ガード | 不正なステータス遷移（例: draft→posted）をAPI層で拒否する | Must |
| F-05-5 | 履歴表示 | 記事ごとのステータス変更履歴を表示 | Should |

### 3.6 コピー用プレビュー（F-06）

| ID | 機能 | 内容 | 優先度 |
|---|---|---|---|
| F-06-1 | note貼り付け用整形 | 記事本文をnoteに貼り付けやすい形式で表示（見出し・本文・ハッシュタグ） | Must |
| F-06-2 | 部分コピー | タイトル / 本文（無料部分・有料部分別）/ ハッシュタグ / 宣伝文 を個別にワンタップコピー | Must |
| F-06-3 | コピー記録 | コピー実行時に copied ステータスへ自動更新 | Must |
| F-06-4 | 有料境界の明示 | 有料記事では「ここから有料」の境界を明示表示 | Must |

### 3.7 売上メモ（F-07）

| ID | 機能 | 内容 | 優先度 |
|---|---|---|---|
| F-07-1 | 売上記録 | 記事ごとに売上（日付・件数・金額・メモ）を手動記録（`revenue_logs`） | Must |
| F-07-2 | 集計表示 | 記事別・月別の合計を表示 | Should |
| F-07-3 | 売上分析 | 傾向分析・改善提案 | Later |

### 3.8 設定（F-08）

| ID | 機能 | 内容 | 優先度 |
|---|---|---|---|
| F-08-1 | プロフィール設定 | 経験・得意分野・発信ジャンル・文体の好みを登録（AI生成に反映） | Must |
| F-08-2 | AI設定 | AIモデル・生成トーンなどの設定（AI API接続を差し替え可能な設計） | Should |
| F-08-3 | データ管理 | 記事データのエクスポート（JSON） | Later |

## 4. 非機能要件

| 分類 | 要件 |
|---|---|
| UI/UX | スマホファースト。主要操作は片手・ワンタップで完結。PWA対応（ホーム画面追加・オフライン閲覧） |
| パフォーマンス | 一覧・詳細表示は1秒以内。AI生成中はプログレス表示とキャンセル可能なUI |
| 可用性 | AI API障害時も記事の閲覧・編集・コピーは可能（AI機能とCRUDを分離） |
| セキュリティ | APIキーはサーバーサイドのみで保持。Supabase RLSによるデータ保護（マルチユーザー化を見据える） |
| 拡張性 | AIプロバイダ差し替え可能な抽象化層。note API連携用のフィールド・ステータスを予約 |
| 監査性 | ステータス遷移・AI生成のログを保持 |

## 5. 技術構成

| 層 | 技術 | 備考 |
|---|---|---|
| フロントエンド | Next.js (App Router) + TypeScript + Tailwind CSS | スマホファースト、PWA対応 |
| バックエンド | Next.js API Routes / Server Actions | |
| DB | Supabase (PostgreSQL) | 将来のマルチユーザー・認証に対応 |
| ORM | Prisma | |
| AI | 抽象化レイヤー経由で接続（MVPではモック or Claude API） | プロバイダ差し替え可能に |
| ホスティング | Vercel（想定） | |

### AI接続の設計方針

- `lib/ai/` に `AIProvider` インターフェースを定義し、`generateThemes()` / `generateArticle()` / `generateSalesPlan()` 等を抽象化
- MVPでは環境変数でモック実装 ⇔ 実API実装を切り替え可能にする

## 8. MVP範囲と将来拡張

### 8.1 MVPに含む

- テーマ提案 / 記事作成フォーム / 無料記事生成 / 有料記事生成
- 記事一覧 / 記事詳細 / 承認ステータス管理
- コピー用プレビュー / 売上メモ / スマホ対応UI（PWA）

### 8.2 MVPに含まない（将来拡張）

- note投稿連携（API公開状況・規約を踏まえて判断）
- 画像生成・サムネイル生成
- SNS宣伝文自動生成（※宣伝文テキスト自体はF-04-1でMVPに含む）
- 売上分析・記事改善提案
- 定期投稿スケジュール / SEO分析
- マルチユーザー・認証

## 9. リスクと対応

| リスク | 対応 |
|---|---|
| noteの規約変更・API仕様 | 連携はMVP外。コピー方式なら規約リスクなし |
| AI生成コンテンツの品質・正確性 | 承認ワークフローで必ず人間の確認を挟む（R2） |
| AI APIコスト | 生成ログでトークン量を記録、モック実装で開発コストゼロ |
| 誤って未承認記事を投稿扱いにする | ステータス遷移ガードをAPI層で強制（F-05-4） |
