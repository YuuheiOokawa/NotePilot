# 08. 追加要件: note投稿戦略・品質管理

- 追加日: 2026-07-07

## 1. テーマ別の連続記事作成（シリーズ）

- `theme_groups`（テーマ）+ `series_items`（計画の各記事）を追加
- `/series` でテーマを入力→AIが**10本前後のシリーズ計画**を提案→ユーザーが役割・価格・タイトルを編集して保存
- 各項目の役割（`SeriesRole`）: `free`(読者獲得) / `paid`(具体ノウハウ・テンプレート) / `summary`(シリーズ導線) / `promo`(有料noteへの導線)
- `/series/[id]` から1本ずつAI生成。生成記事には `themeGroupId` / `seriesNumber` / `seriesRole` が紐付く

## 2. 無料・有料の配分

- AI提案時の配分ルール: **無料6〜7本 / 有料2〜3本 / まとめ1本 / 宣伝1本**
- 無料=信頼獲得重視、有料=手順・テンプレート・チェックリスト・実体験重視（プロンプト・モック両方に反映）
- 「同じ内容の記事を量産しない」→ 品質チェックの重複検出＋シリーズ計画で切り口を分ける設計

## 3. 投稿スケジュール管理

- 記事に `scheduledAt`（投稿予定日時）と `postFrequencyGroup`（時間バケツ 例:`2026-07-07T10`）を追加
- **1時間に最大3投稿**: `PATCH /api/articles/[id]/schedule` が同一時間バケツの件数を数え、4件目は409で拒否（`lib/schedule.ts`）
- `/schedule`（投稿キュー画面）: 投稿予定キュー / 時間帯ごとの枠使用状況 / 承認待ち / 承認済み未スケジュール / 投稿済み
- ⚠️ スケジュールは**管理用の目安**。実際の投稿はユーザー承認後に手動（自動スパム投稿はしない）

## 4. 品質チェック機能

- `POST /api/articles/[id]/quality-check` が AI（モック時はルールベース）でチェックし、`quality_checks` に履歴保存
- チェック項目: 誤字脱字 / 文法 / 表現の不自然さ・誇大表現 / 内容の重複 / タイトルと本文の一致 / 無料・有料の切り分け / 読者価値 / 未確認情報 / 出典が必要な情報
- 結果は記事の `qualityScore` / `typoCheckStatus` / `factCheckStatus` に反映
- **本文を編集するとチェック結果は無効化**され、再チェックが必要になる（作成後必ずチェックさせる設計）

## 5. ファクトチェック方針

- 数値・法律・制度・試験情報・料金などの記述を抽出し、`fact_claims` に**要確認フラグ付き**で保存
- ユーザーが1件ずつ「確認済み」「本文から削除した」に変更できる（`PATCH /api/claims/[id]`）
- `hasUnverifiedClaims` が true の間は `publishReadinessStatus` が絶対に `ready` にならない（`lib/review.ts`）
- **未確認情報が残っている記事は承認（review→approved）できない**: API層の承認ガードで409拒否
- AIプロンプトにも「断定表現を避ける」「不明情報は要確認として列挙」を明記

## 6. 記事レビュー画面

記事詳細に `ReviewPanel` を追加:

- 品質チェック結果（スコア / 誤字・表現 / ファクトの3ステータス）
- チェック結果詳細（誤字脱字・誇大表現・重複・タイトル一致・切り分け・読者価値）
- 要確認リスト（カテゴリ・理由付き、確認済み切り替え）
- 改善提案
- レビューメモ（`reviewNotes`）
- 投稿可否ステータス（チェック未実施 / 要修正・要確認 / **投稿準備OK**）
- 投稿予定設定（1時間3件の枠チェック付き）

## 7. 追加したDB項目（note_articles）

`themeGroupId` / `seriesNumber` / `seriesRole` / `isPaid` / `scheduledAt` / `postFrequencyGroup` / `qualityScore` / `typoCheckStatus` / `factCheckStatus` / `hasUnverifiedClaims` / `reviewNotes` / `publishReadinessStatus`

新テーブル: `theme_groups` / `series_items` / `fact_claims` / `quality_checks`

## 8. 重要ルールの実装ポイント

| ルール | 実装 |
|---|---|
| AIが勝手に投稿しない | 投稿処理自体が存在しない。postedはユーザーの手動記録のみ |
| 投稿は必ずユーザー承認制 | review→approved はユーザー操作のエンドポイントのみ＋遷移ガード |
| 未確認情報がある記事は投稿可能にしない | `canApprove()` で承認を409拒否＋UI側もボタン無効化 |
| 有料記事は明確に価値が高い内容に | 生成・チェック両方のプロンプトで手順/テンプレート/実体験を要求 |
| 同じ内容を量産しない | シリーズ計画で切り口分割＋重複検出チェック |
| 1時間3投稿は目安・自動スパムなし | スケジュールは管理メモ。実投稿は常に手動 |
